Critical findings:

Reset endpoint does not enforce an active/running session (artifacts/api-server/src/routes/session.ts:260-262, 268-276). It resets the latest session regardless of status. If the latest session is completed/error, this endpoint still rewrites startedAt/stoppedAt and returns success, which conflicts with “reset current session” semantics and can leave a non-running session with a fresh start timestamp.
DB fallback path can silently report success without actually resetting DB data (session.ts:271-283). When db exists and reset fails, code falls back to memStore and still returns 200 + WS session_reset. In DB-backed deployments, mem-store may not contain the authoritative session/readings, so clients think reset succeeded while persistent rows remain unchanged.
Reset operation is non-atomic in DB mode (session.ts:272-277). Readings delete and session update run as separate statements without a transaction. A mid-operation failure can partially apply reset (e.g., readings deleted but session not updated), returning 500 after side effects.
Feature-specific checks requested:

Race with in-flight motor command: no direct motor mutation in reset path (good), but no concurrency guard with session stop/close paths; concurrent updates can still produce inconsistent session timestamps/status.
terminal-bus stale closure/unmount: generally safe in current usage. registerTerminalHandler cleanup prevents dangling handler (terminal-bus.ts:17-21, SystemTerminal.tsx:320-323).

Security: none observed (no serious auth/injection/secret exposure issues introduced in these changes).
